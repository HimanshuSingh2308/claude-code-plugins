# Game Hotfix

Apply an emergency fix to a deployed game with enhanced safety measures.

**Arguments**: $ARGUMENTS

## Overview

Use this command for urgent fixes to production games. It provides:
- Expedited workflow (skip scouting/design phases)
- Enhanced guardrails (extra confirmations)
- Automatic rollback preparation
- Minimal change principle

**When to use**:
- Critical bug affecting players
- Security vulnerability discovered
- Game-breaking issue reported
- Leaderboard corruption

**When NOT to use**:
- New features (use `/weekly-game-release`)
- Major refactors (use `/game-audit --fix`)
- Non-urgent improvements

## Command Options

```
/game-hotfix <game-slug> [options]

Arguments:
  <game-slug>              Game to fix (e.g., "emoji-match")

Options:
  --issue <number>         Link to GitHub issue
  --description "<bug>"    Describe the bug to fix
  --dry-run                Preview fix without applying
  --skip-qa                Skip QA validation (dangerous)
  --no-merge               Create PR only, no auto-merge
```

## Usage Examples

### Fix from Issue
```bash
/game-hotfix emoji-match --issue 42
```

### Fix with Description
```bash
/game-hotfix word-stack --description "Score not saving after level 5"
```

### Preview Fix
```bash
/game-hotfix tetris --dry-run --description "Pieces overlap at high speed"
```

## Hotfix Workflow

### Step 1: Pre-Flight Checks

```yaml
Verify:
  - Game exists in apps/web/src/games/{game-slug}/
  - Main branch is clean
  - No pending PRs for this game
  - User has push permissions
```

### Step 2: Create Hotfix Branch

```bash
git checkout main
git pull origin main
git checkout -b hotfix/game-{game-slug}-{timestamp}
```

### Step 3: Understand the Bug

```yaml
IF --issue provided:
  Fetch issue details from GitHub
  Extract: description, reproduction steps, expected behavior

IF --description provided:
  Parse bug description

THEN:
  Read game source code
  Identify affected files
  Analyze root cause
```

### Step 4: Implement Fix

```yaml
Principles:
  - Minimal change (smallest fix possible)
  - No refactoring
  - No new features
  - Preserve existing behavior

Guardrails (ENHANCED):
  BLOCKED:
    - Any file outside games/{game-slug}/
    - api-client.js, auth.js
    - index.html, sitemap.xml (for hotfix)
    - Any deletion

  CONFIRM (every change):
    - "Modify {file}? Change: {summary}"
    - Show diff preview before confirmation

  SAFE:
    - Read any file
    - Run tests
```

### Step 5: Validate Fix

```yaml
Required:
  - npm run lint (must pass)
  - npm run build (must pass)

Recommended (unless --skip-qa):
  - Invoke game-qa-tester on changed files
  - Verify bug is fixed
  - Check for regressions
```

### Step 6: Create Hotfix PR

```yaml
Branch: hotfix/game-{game-slug}-{timestamp}
Title: "hotfix({game-slug}): {brief description}"

Body:
  ## Emergency Hotfix

  **Game**: {game-name}
  **Issue**: #{issue_number} (if linked)
  **Severity**: CRITICAL

  ### Bug Description
  {description}

  ### Root Cause
  {analysis}

  ### Fix Applied
  {summary of changes}

  ### Files Modified
  {list}

  ### Testing
  - [ ] Lint passes
  - [ ] Build passes
  - [ ] Bug verified fixed
  - [ ] No regressions found

  ### Rollback
  If issues arise, revert with:
  \`git revert {commit_sha}\`

  ---
  Generated by Game Hotfix Command
```

### Step 7: Merge (if not --no-merge)

```yaml
CONFIRM: "Merge hotfix PR #{n} to main? This is a production deployment. [Y/n]"

IF confirmed:
  gh pr merge {pr_number} --squash --delete-branch

  # Verify deployment
  WAIT 30 seconds
  CHECK: Game loads without errors
```

## Enhanced Guardrails

Hotfix mode has **stricter guardrails** than normal development:

### Scope Restrictions

```yaml
ALLOWED files (for hotfix):
  - apps/web/src/games/{game-slug}/**

BLOCKED files (for hotfix):
  - apps/web/src/games/{other-games}/**
  - apps/web/src/index.html
  - apps/web/src/sitemap.xml
  - apps/web/src/leaderboard/index.html
  - apps/web/src/js/**
  - apps/api/**
  - Any config files
```

### Change Limits

```yaml
Max files modified: 3
Max lines changed: 100
Max new files: 0 (hotfix should not add files)

IF exceeded:
  WARN: "Hotfix scope exceeded. Consider using /weekly-game-release instead."
  CONFIRM: "Proceed anyway? [Y/n]"
```

### Mandatory Confirmations

Every change requires explicit confirmation:
```
CONFIRM: "Modify games/{slug}/index.html?"
  Lines affected: {start}-{end}
  Change summary: {description}

  [Y] Apply change
  [n] Skip
  [d] Show full diff
  [a] Abort hotfix
```

## Rollback Preparation

Before merge, the command:

1. **Records rollback command**:
   ```bash
   # To rollback this hotfix:
   git revert {merge_commit_sha}
   git push origin main
   ```

2. **Creates rollback PR draft** (not submitted):
   - Title: "revert: Rollback hotfix for {game-slug}"
   - Body: Pre-filled with revert details

3. **Saves state**:
   - `~/Documents/weekly-games/hotfixes/{game-slug}-{timestamp}.json`

## Error Handling

### If Fix Fails Validation

```yaml
Action: Abort before merge
Cleanup:
  - Delete hotfix branch (local)
  - Do NOT push or create PR

Message:
  "Hotfix validation failed. Branch deleted."
  "Error: {validation_error}"
  "Please fix manually or retry with corrected approach."
```

### If Merge Fails

```yaml
Action: Keep PR open
Message:
  "Merge failed: {error}"
  "PR #{n} remains open for manual resolution."
  "URL: {pr_url}"
```

### If Post-Merge Issues

```yaml
IF game errors detected after merge:
  ALERT: "Post-merge issues detected!"
  SHOW: Rollback instructions
  CONFIRM: "Execute rollback? [Y/n]"
```

## Notes

- Hotfixes bypass the full release workflow for speed
- Enhanced guardrails compensate for reduced review
- Always verify fix in production after merge
- Use `--skip-qa` only for true emergencies
- Consider follow-up PR for proper testing/refactoring
